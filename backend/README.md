# Backend «Академического Салона»

Воспроизводимая серверная основа бота и same-origin API. В репозитории нет
рабочей базы, токенов, логов, клиентских payload-файлов и виртуального окружения.

## Что является источником правды

- `catalog/pricing.v1.json` — versioned-каталог цен и результатов услуги.
- `migrations/*.sql` — последовательные миграции SQLite.
- `.env` на сервере — секреты и параметры окружения; файл никогда не коммитится.
- Юридические редакции пока заданы в `app/config.py::DOC_EDITIONS`; их нельзя
  менять отдельно от опубликованных документов.

Публичный контракт каталога доступен по `GET /api/catalog/pricing`. Сайт должен
использовать этот JSON или проходить parity-тест с ним. `base`, `turn` и `vip`
означают соответственно `diagnostic`, `editing` и `support`; это разные
результаты с явными ценами, не множители маркетингового названия.

В Telegram-визарде результат выбирается явно до расчёта цены: «Диагностика»,
«Редакторский аудит» или «Сопровождение». Затем клиент отмечает, какие
собственные материалы уже есть. Старые deep-link-и и незавершённые FSM-сессии
без tier не ломаются: бот просит сделать выбор и только после этого разрешает
создать заказ.

## Локальный запуск

Требуется Python 3.9+; на production также нужны `sqlite3`,
`fonts-dejavu-core` и LibreOffice (`soffice`) для безопасной конвертации
DOC/DOCX и предпросмотров.

```bash
cd backend
python3 -m venv .venv
.venv/bin/python -m pip install --upgrade pip
.venv/bin/python -m pip install -r requirements.lock
cp .env.example .env
```

Заполните `.env` локально. Не пересылайте его в чат и не помещайте в Git.
Процесс запускается штатной точкой приложения:

```bash
set -a
. ./.env
set +a
.venv/bin/python -m app.bot
```

Проверка без запуска Telegram:

```bash
PYTHONPATH=. .venv/bin/python -m unittest discover -s tests -v
```

## Обязательный порядок production-деплоя

1. Остановить запись в приложение или перевести его в maintenance.
2. Зафиксировать текущий commit и состояние сервиса.
3. Создать online-backup SQLite встроенной командой `.backup`:

   ```bash
   install -d -m 700 /var/backups/akademsalon
   stamp="$(date -u +%Y%m%dT%H%M%SZ)"
   sqlite3 "$DB_PATH" ".timeout 5000" \
     ".backup '/var/backups/akademsalon/salon-${stamp}.sqlite3'"
   sqlite3 "/var/backups/akademsalon/salon-${stamp}.sqlite3" \
     "PRAGMA quick_check;"
   sha256sum "/var/backups/akademsalon/salon-${stamp}.sqlite3" \
     > "/var/backups/akademsalon/salon-${stamp}.sqlite3.sha256"
   ```

   Продолжать можно только если `quick_check` вернул `ok`.

4. Развернуть код в новый release-каталог, создать новое venv и установить
   `requirements.lock`. `requirements.txt` содержит только прямые зависимости
   и используется при осознанном обновлении lock-файла, не в production.
5. Запустить тесты против копии базы. Первый `db.init()` применит только
   отсутствующие миграции и запишет их в `schema_migrations`.
6. Проверить:
   - `SELECT * FROM schema_migrations ORDER BY version;`
   - `PRAGMA foreign_key_check;`
   - `PRAGMA quick_check;`
   - `GET /api/catalog/pricing`;
   - вход, истечение сессии и модерацию отзыва без согласия (ожидается HTTP 409).
7. Переключить systemd на новый release, запустить и проверить журнал без
   вывода environment.

Перед переключением включите безопасный log format и сначала Report-Only CSP из
`ops/nginx-security.conf.example`. После browser-smoke переведите CSP в
enforce-режим.

Миграция `0001` переводит прежние публичные отзывы в `pending`, потому что в
старой схеме нет доказательства отдельного согласия. Тексты не удаляются.
Публиковать их повторно можно только после нового согласия клиента.

Нельзя накатывать миграции на единственную копию базы и нельзя запускать новый
код, пока backup не проверен.

## Rollback

Миграции forward-only. Не удаляйте добавленные колонки вручную.

- Если новый код не стартовал и миграции не применялись: переключите systemd на
  предыдущий release и перезапустите.
- Если миграции применились, но записи новым кодом ещё не принимались:
  остановите сервис, сохраните аварийную копию текущей базы, восстановите
  проверенный pre-deploy backup во временный файл и атомарно замените DB.
- Если после миграции уже были новые заказы, оплаты, сообщения или отзывы:
  простой restore потеряет данные. Оставьте базу на месте, откатите только код
  при подтверждённой обратной совместимости либо проведите отдельную сверку и
  перенос новых строк. Автоматический destructive rollback запрещён.

После любого rollback снова выполните `PRAGMA quick_check`,
`PRAGMA foreign_key_check` и smoke-тесты.

## Защитные инварианты

- Каждая новая заявка, в том числе из авторизованного кабинета, требует
  `consent: true` и точную текущую строку `consent_doc`, связанную с отдельным
  `consent-request.html`. Аккаунт не заменяет согласие; его время ставит сервер.
- Отзыв нельзя перевести в `approved` без отдельного согласия, его времени и
  версии текста согласия. Ограничение проверяет сервис и SQLite-триггер.
- Имя автора публикуется только если соответствующая категория согласия включена.
- Сессии имеют idle и absolute TTL; impersonation-сессии короче.
- Session tokens хранятся как SHA-256 digest; migration сохраняет действующие
  клиентские сессии, но убирает готовые Bearer-ключи из SQLite.
- Session token принимается только в `Authorization: Bearer …`, не в URL.
- Гостевой доступ к одному делу передаётся в `X-Order-Token`, список дел —
  в `X-Order-Tokens` (значения через запятую). Query `?token=`/`?tokens=`
  поддерживается временно только для миграции старых вкладок и не должен
  использоваться новым frontend.
- IP посетителя больше не передаётся внешнему `ip-api.com`; геометка админки
  по умолчанию отключена.
- First-party `visits` хранит только случайный browser id, IP, user-agent,
  очищенную страницу/источник, агрегатный шаг и время. `user_id`, `order_id` и
  `contact` не принимаются слоем записи; legacy-связки обнуляются на старте.
  Сырые визиты, IP и geo-cache старше 365 дней удаляются автоматически.
- `app/services/intake_guard.py` одинаково проверяет пользовательские поля
  Telegram-визарда и `POST /api/orders`. Явный запрос выполнить аттестацию или
  тест вместо клиента, выдумать данные/источники либо обойти проверку получает
  `422 request_outside_scope` и не записывается как коммерческий заказ.
  Негированные формулировки и образовательные вопросы покрыты regression-
  тестами; договорные описания из cart не сканируются как запрос клиента.
- Реальные идентификаторы администраторов и Telegram-чатов не имеют defaults и
  задаются только окружением.

## Секреты гостевого доступа

`orders.access_token` хранится как Fernet-ciphertext с префиксом `e1$`, а
lookup выполняется по отдельному `d1$` SHA-256 digest. При первом старте
legacy plaintext-значения шифруются без смены уже отправленных клиентам ссылок.

Перед первым запуском этой версии обязательно задайте один постоянный
`ORDER_ACCESS_TOKEN_KEY` (корректный Fernet-ключ) и сделайте защищённую копию
SQLite. Ключ нельзя менять или терять: без него существующие гостевые ссылки
не расшифруются. Ключ не коммитится, не логируется и хранится отдельно от
BOT_TOKEN, платёжных и почтовых секретов.

Дополнительные правила остаются обязательными:

- не включать `$request` или `$args` с секретами в nginx access log;
- не отправлять ссылки кабинета в стороннюю аналитику;
- сохранять `Referrer-Policy: no-referrer` на API-ответах;
- хранить backup базы с правами `0600`;
- при подозрении на утечку вращать `access_token` конкретного заказа.

Session token также хранится только в хешированном виде.
